**Checklist**
1. Parsing correctness: yes. The fields `validate.py` reads match `InferenceResult`.
2. Sanity-check semantics: mostly yes. The F-vs-C prompt comparison is matched on `(question_id, model_tag, rep_number)`, which is correct for the current runner’s paired-trace logic.
3. ITT: yes. Extraction failures remain counted as incorrect; only the status message is wrong.
4. Delta direction: yes. The subtraction directions are right; `B-A`, `B-C`, `C-F`, and `B-O` are all signed so positive means the intended advantage. The only nit is the misleading `C_vs_B` label for a `B-C` delta.
5. Critical vs minor exit logic: yes in `validate.py`; no for the shell’s custom banner path.
6. Runner args: the actual invocation flags are correct for validation. The printed recovery command for trace-bank generation is not.
7. `--min-reps`: it does not currently matter because the parameter is unused. If enforced, it should be tied to expected validation coverage, not left at 2 for a 4-rep run.
8. Inference-run non-zero exit: the shell aborts via `set -e`, so it does stop, but not gracefully.
9. Missing checks: yes. The biggest gaps are run scoping, expected coverage, duplicate-row detection, seed/pairing validation, and model/version consistency.
10. Pre-registered contrasts: no. It only covers the phase-1-testable subset (`B-C`, `C-F`, `B-O`) plus optional `B-A` if A happens to be in the log.
11. Condition I sub-inference check: not needed for the current `B/C/F/O` wrapper, but needed if `validate.py` is ever reused on smoke-test or confirmatory logs, because current runner logs both I sub-inferences and the parent majority-vote record under condition `I`.
12. Mixed-condition / mixed-run logs: high contamination risk.
13. Zero-inference condition: currently dangerous. A missing condition simply disappears from checks and comparisons and can still yield PASS.
