Here is the audit report for the thinking-token experiment fixes, categorized by issue area with severity ratings and file/line citations.

### FIX 1: Token Counting (trace_bank.py, condition_builder.py)

**1. Proxy Tokenizer (`cl100k_base`)**
* **Rating:** MINOR
* **Analysis:** `cl100k_base` is a reasonable proxy for *consistency* (ensuring two strings matched to the same token count are roughly the same length). However, Qwen 3.5 (~152k vocab) and Gemma 4 (256k vocab) have larger vocabularies, meaning `cl100k_base` will systematically overcount tokens relative to the models' native tokenizers by roughly 10-20%. Because the protocol only requires length-matching between experimental conditions (e.g., C vs J), absolute count accuracy is less critical than determinism.

**2. Semantic Destruction vs. Token Count**
* **Rating:** IMPORTANT
* **Analysis:** Shuffling BPE token IDs perfectly destroys semantic coherence, but it does *not* cleanly decode back to standard text. BPE tokens represent byte chunks, including continuation bytes. Shuffling them randomly will produce invalid UTF-8 byte sequences, which `tiktoken.decode()` will replace with the Unicode replacement character (``). 

**3. The `count_tokens(shuffled_text) == len(token_ids)` Assertion**
* **Rating:** CRITICAL
* **Citation:** `trace_bank.py:214`
* **Analysis:** This assertion **will frequently fail** and crash the experiment. The `encode → shuffle → decode → re-encode` process is not bijective for two reasons:
  1. **Greedy Merging:** If token ` A` and token `B` happen to be shuffled adjacent to each other, re-encoding ` A B` might greedily merge them into a single token ` AB`, decreasing the total token count.
  2. **Invalid Bytes:** Because invalid UTF-8 sequences are replaced by `` during `decode()`, the re-encode phase will map all those broken byte combinations to the standard replacement character token, radically altering the token count.

**4. Filler Generation Accuracy**
* **Rating:** MINOR
* **Citation:** `condition_builder.py:186`
* **Analysis:** `generate_filler()` approximates the target count by popping full words. Because words do not align cleanly with token boundaries, it cannot guarantee exact BPE token matches. Condition J will likely be slightly shorter than Condition C by 1-3 tokens. This is acceptable for a bulk length-control, but violates the strict "identical token counts" claim in the docstring.

**5. Trace Bank Legacy Field Mismatch**
* **Rating:** CRITICAL
* **Citation:** `trace_bank.py:75` (inside `_load()`)
* **Analysis:** Existing trace banks on disk store `token_count` generated by the old word-count logic. When `TraceBank._load()` reads them, it blindly loads this integer. Later, Condition J calls `generate_filler(entry.token_count)`. Because `entry.token_count` is a word count (e.g., 100), `generate_filler()` will produce 100 *BPE tokens* of filler—which is only ~70 words. Condition J will be systematically ~30% shorter than the B-traces it is supposed to match.
* **Proposed Fix:** Recalculate the BPE count dynamically upon load to seamlessly upgrade legacy trace banks.
  ```python
  entry = TraceEntry(**data)
  entry.token_count = count_tokens(entry.thinking_tokens) # Add this line
  ```

### FIX 2: Condition I (run_experiment.py)

**6. The `k` Computation Hardcode**
* **Rating:** IMPORTANT
* **Citation:** `run_experiment.py:210`
* **Analysis:** Estimating the no-think response length with a hardcoded `mean_a_tokens = 50` is dangerous. If a specific domain (like math) naturally requires 200 tokens for the final answer, `k` will be artificially inflated by 4x, giving Condition I a massive, unfair compute budget advantage over Condition B.

**7. Sub-inference Seed Collision**
* **Rating:** MINOR
* **Citation:** `run_experiment.py:217`
* **Analysis:** `sub_seed = seed * 1000 + i`. Since base `seed` comes from `SEED_GRID` (1-12), `sub_seed` spans 1000–12019. This safely avoids collisions with the standard 1-12 seeds used by other conditions.

**8. Double Counting Sub-inferences**
* **Rating:** CRITICAL
* **Citation:** `run_experiment.py:224` and `257`
* **Analysis:** Both the individual sub-inferences AND the parent majority-vote result are logged with `condition="I"`. Any downstream analysis script that groups by `condition == "I"` will aggregate $k+1$ rows per repetition, completely corrupting Condition I's accuracy, duration, and token count averages.
* **Proposed Fix:** Change the sub-inferences to log as `condition="I_sub"`.

**9. `source_trace_id` Reuse**
* **Rating:** IMPORTANT
* **Citation:** `run_experiment.py:222`
* **Analysis:** Reusing `source_trace_id` as a parent link for sub-inferences is a hack that pollutes the schema. `source_trace_id` elsewhere means "a link to a TraceBank B-condition trace." Downstream scripts attempting to look up `sub.source_trace_id` in the TraceBank will fail. A dedicated `parent_inference_id` field should be added to the `InferenceResult` dataclass.

**10. `grade()` and Answer Types**
* **Rating:** MINOR
* **Citation:** `run_experiment.py:236`
* **Analysis:** The `_majority_vote` uses exact string matching on the elements of `answers`. Because `extract_answer` (which applies `normalize_answer`) is run on the sub-inferences before they are appended to the `answers` list, the vote correctly takes place over *normalized* strings. Sub-inferences with `extraction_failed=True` are safely excluded.

### FIX 3: Condition N (condition_builder.py, run_experiment.py)

**11. Temperature=0 Deviation from Protocol**
* **Rating:** IMPORTANT
* **Citation:** `condition_builder.py:72`
* **Analysis:** The plan calls for an "empty-trace / forced minimal trace" to see if the model can bypass CoT while in think mode. `temperature=0.0` does *not* force a minimal trace; it forces a deterministic trace. The model will likely still generate a massive, standard CoT, just without sampling variance. This fails to execute the required protocol test. 

**12. Option Merging Logic**
* **Rating:** MINOR
* **Analysis:** The option merging logic in `ollama_client.py` (`merged_options.update(options)`) behaves correctly and will successfully override `DEFAULT_OPTIONS["temperature"]` to 0.0 when Condition N passes it.

**13. Repetition Redundancy**
* **Rating:** IMPORTANT
* **Citation:** `run_experiment.py:166`
* **Analysis:** Greedy decoding (`temperature=0.0`) renders the `seed` irrelevant. Running Condition N for 12 repetitions (using seeds 1-12) will produce exactly identical outputs 12 times per question, wasting massive amounts of compute. Condition N should ideally be locked to a single rep.

### GENERAL

**14. Import Cycles**
* **Rating:** MINOR
* **Analysis:** There are no strict circular imports. `run_experiment.py` imports `trace_bank` and `condition_builder`. `condition_builder` imports `count_tokens` from `trace_bank`. `trace_bank` only imports third-party or standard libraries. Local imports inside functions (e.g., `condition_l`) defer resolution safely. 

**15. Alias Backward Compatibility**
* **Rating:** IMPORTANT
* **Citation:** `trace_bank.py:218`
* **Analysis:** The alias `shuffle_tokens_word_level = shuffle_tokens` maintains API compatibility (callers won't throw a `NameError`), but fundamentally changes behavioral semantics. Code expecting semantic word-chunks to be shuffled will now get raw BPE-shuffling (which can tear words apart and break UTF-8 sequences). If older data-prep scripts rely on this alias, their output will change drastically.

**16. TraceBank Write Race Conditions**
* **Rating:** CRITICAL
* **Citation:** `trace_bank.py:78` (`save()`) vs `62` (`add_trace()`)
* **Analysis:** `add_trace()` uses `O_APPEND` which is generally safe across processes. However, `run_trace_bank_generation()` finishes by calling `trace_bank.save()`, which uses `"w"` mode. If the script is run concurrently (e.g., across multiple nodes or batch jobs writing to a shared file system), one process calling `save()` will instantly truncate the shared `trace_bank.jsonl` file, destroying any traces currently being written by other concurrent processes.
