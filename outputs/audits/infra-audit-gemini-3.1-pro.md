Here is the infrastructure audit of the Thinking-Token Experiment Runner, evaluated against the experimental protocol and system robustness requirements.

### Audit Findings

| Severity | File | Finding | Recommendation |
| :--- | :--- | :--- | :--- |
| **CRITICAL** | `ollama_client.py`, `run_experiment.py` | **Premature logging causes total data loss:** `OllamaClient.chat` and `generate_raw` append the `InferenceResult` to the JSONL log immediately before returning. However, `score_inference()`—which populates `correct`, `extracted_answer`, `extraction_failed`, and `ground_truth`—is called *after* the client returns. All logs will permanently have empty/null values for grading metrics and trace sourcing metadata. | Move `self._log(result)` out of `OllamaClient` entirely. Call the logging routine in `run_single_inference()` only after all scoring and trace metadata assignments are complete. |
| **CRITICAL** | `run_experiment.py`, `condition_builder.py` | **Chat API breaks prefill continuity:** Prefill conditions (C, D, E, F, etc.) pass a list ending in an `assistant` message to Ollama's `/api/chat` endpoint. The `chat` API terminates the assistant message (appending `<\|im_end\|>`) and prompts a new turn, completely invalidating the prefill methodology (evidenced by the "eval=23" log comment). `generate_raw` was written for this but is unreferenced dead code. | Update `run_single_inference` to route prefill conditions through `get_model_template_tokens` to construct an open-ended raw prompt string, then call `client.generate_raw()` instead of `client.chat()`. |
| **CRITICAL** | `condition_builder.py` | **Missing `<think>` tags in prefilled reasoning:** Ollama strips `<think>` tags when populating `result.thinking_tokens`. When these raw traces are injected back into prefill conditions via `_prefill_messages` (where `think=False`), the model sees the reasoning steps as regular conversational text. This puts the model severely out-of-distribution, confounding the experiment. | Explicitly wrap the injected `trace_text`, `scaffold_text`, and fillers in `<think>\n...\n</think>` tags when formatting the prefill prompt. |
| **CRITICAL** | `run_experiment.py` | **Condition I inferences are completely unlogged:** `_run_condition_i()` aggregates `k` inferences and constructs a final combined `InferenceResult` for the majority vote. Because it bypasses the standard return flow, this final aggregated result is never written to the JSONL log file. | Explicitly call the log writer for the aggregated `result` object before returning it at the end of `_run_condition_i()`. |
| **IMPORTANT** | `condition_builder.py` | **Condition N fails to minimize traces:** Condition N is defined as "Empty-trace think mode — think=true with forced minimal trace." However, `condition_n()` currently just calls and returns `condition_b()` exactly. There is no structural difference or instruction to suppress step-by-step reasoning. | Update `condition_n()` to include a strict system instruction (e.g., *"Do not think step-by-step; output the final answer immediately"*) or manually prefill a closed `</think>` block. |
| **IMPORTANT** | `run_experiment.py` | **Condition L fraction defaults to 1.0 (duplicate of C):** Condition L truncates traces using `question.get("l_fraction", 1.0)`. Because the procedural generators do not output an `l_fraction` parameter, it universally defaults to `1.0`. This makes Condition L functionally identical to Condition C. | Add an `--l-fraction` CLI argument, or build a dedicated sweep over fraction sizes when running Condition L. |
| **IMPORTANT** | `trace_bank.py` | **Trace matching tolerance is too restrictive:** `get_wrong_question_trace()` enforces a strict `±5%` (`tolerance=0.05`) token count match limit for Condition G. Given a constrained pool of ~260 questions, finding a same-domain match with such strict length bounds (and <20% word overlap) will frequently fail, causing the replicate inference to crash. | Relax the default `tolerance` to at least `0.15` - `0.20`, or implement a progressive fallback loop that expands the tolerance band if no candidates match. |
| **MINOR** | `score.py` | **Brittle MCQ extraction regex:** The regex `r"^([A-D])\)?\.?\s"` requires a trailing space (`\s`). Because `last` is `.strip()`ped prior to regex search, bare outputs like `"B."` will fail extraction. Additionally, `normalize_answer()` leaves prefixes like `"Option A"`, causing `_extract_mcq_letter` to fail. | Change the regex space constraint to permit end-of-string: `(?:\s\|$)`. Update `normalize_answer()` to safely strip `"Option "` prefixes. |

### Publication Readiness
**Assessment: NOT READY**

The repository contains critical structural flaws that completely invalidate its primary experimental mechanism. By utilizing standard conversational endpoints (`/api/chat`) for assistant prefilling, the protocol inadvertently terminates the model's cognitive block rather than prompting it to continue. Combined with the absence of `<think>` tag wrappers around injected traces and the critical logging sequence errors that result in total loss of grading data, the framework cannot produce scientifically usable data in its current state. The infrastructure requires a structural refactor to raw prompt generation and corrected logging pipelines before Phase 1 or 2 can commence.
